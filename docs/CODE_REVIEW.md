# 코드 리뷰 기록

## 리뷰 정보

- 기준일: 2026-07-18
- 기준 버전: 1.0.9
- 범위: 전투 입력·시뮬레이션·결과 처리, 영구 상태와 저장, JSON 설정 계층, 전투 화면 구성, 테스트와 확장성
- 심각도 기준: P1은 즉시 배포를 막아야 하는 치명 문제, P2는 다음 기능 확장 전에 해결할 정확성 문제, P3는 계획적으로 줄일 구조·성능 부채입니다.

## 결론

P1 문제는 발견되지 않았고, 리뷰 중 확인한 P2 정확성 문제는 모두 수정했습니다. 현재 한 마리 보스 전투, 난이도별 맵·보스 선택, 전투 화면 탭 공격, 저장·오프라인 생산과 4대 메뉴 흐름은 자동 테스트 범위에서 동작합니다. 전투 화면, 엔진, selector, 리듀서와 설정 검증을 책임별로 분리해 새 기능이 한 파일에 계속 누적되지 않는 구조로 정리했습니다.

다만 “향후 다수 개체·상태 효과를 추가해도 성능 문제가 없다”고 미리 판단할 단계는 아닙니다. 현재 단일 보스 구조를 넘어서기 전에 동일 기기 기준의 프로파일링과 개체 수 상한을 결정해야 합니다.

- 15개 난이도는 `difficulties.json → enemy-waves.json → monsters.json`을 통해 각각 고유 보스 하나를 선택합니다.
- `battle-maps.json`은 각 난이도 ID와 고유 전장 하나를 연결합니다.
- 15종 보스의 기준 전투 프로필을 동일하게 유지해 에셋·이름 추가가 현재 밸런스를 바꾸지 않도록 했습니다.
- 정적 이미지 키 매핑은 `BattleMapImage.ts`와 `MonsterSprite.tsx`로 한정했습니다.
- 설정 테스트가 난이도별 맵·웨이브·보스 ID의 유일성과 참조 무결성을 확인합니다.

### 전투 화면 전체 탭과 전파 격리

활성 전투의 전장 `Pressable`만 성 원반 명령을 받습니다. 거대 원반 버튼과 쿠키 성 자체 입력은 `stopPropagation()`으로 상위 전장 탭과 분리되며, 거대 원반 버튼은 사용할 수 없는 상태에서도 이벤트 전파를 막습니다. 결과 모달과 하단 메뉴는 전장 입력 영역 밖에 있어 한 번의 탭이 전투 명령과 메뉴 명령을 동시에 만들지 않습니다.

엔진의 기존 쿨타임·사거리·소유 검증은 그대로 적용되므로 화면 빈 곳을 눌렀다는 이유만으로 조건을 우회하지 않습니다. 이 레이어 구성 자체에서 확인된 UI 부작용은 없습니다. 거대 원반 버튼도 엔진 명령 하나만 호출하고 상위 전장 탭과 분리됩니다.

### 엔진 입력 안정성 보강

- 타이머와 탭 명령을 최신 `stateRef`에서 직렬화하고, 순수 `tryThrowCastleDisc`·`tryThrowGiantDisc` 전이가 현재 상태의 전투 진행 여부·쿨타임·사거리를 다시 검증합니다.
- 같은 렌더 사이의 빠른 성 공격 두 번은 첫 발이 반영된 최신 쿨타임에서 두 번째 요청을 거부합니다.
- 수동 발사체 ID는 시각뿐 아니라 단조 증가 순번을 포함해 같은 밀리초 입력도 충돌하지 않습니다.
- 단조 증가 이벤트 ID 커서로 `onEvent` 콜백 참조만 바뀐 경우에도 이미 소비한 이벤트를 다시 전달하지 않습니다.
- 프레임 경과 시간은 `0~maxDeltaMs`로 제한해 기기 시각이 뒤로 이동해도 보스나 원반이 역행하지 않습니다.

이 보강은 전투 피해·쿨타임·속도·난이도 수치를 변경하지 않습니다.

### 전투 이벤트 저널·연출 분리

- `BattleState.lastEvent` 단일 슬롯을 제거하고, 화면 연출용 `events`는 최근 상한만 유지하며 아직 전달하지 않은 사건은 별도 `pendingEvents`에 손실 없이 보존합니다. `eventSequence`는 두 저장소와 독립적으로 단조 증가합니다.
- hook은 마지막으로 소비한 ID 이후의 pending을 순서대로 전달한 뒤 해당 ID까지만 명시적으로 acknowledge합니다. 단일 `advanceBattle`이 표시 상한을 넘는 이벤트를 만들어도 효과음이 누락되거나 중복되지 않습니다.
- reset·start는 미전달 pending과 시퀀스를 그대로 승계합니다. `onEvent`는 TypeScript 계약상 필수이며, 런타임에서 누락된 비정상 경우에는 해당 기간 pending을 명시적으로 버려 메모리가 무한히 늘지 않고, 나중에 콜백이 생겨도 과거 효과를 재생하지 않습니다.
- 화면은 전체 전달 저널과 별개인 `presentationEvent`를 사용합니다. `victory`·`defeat`가 마지막 피격 이벤트를 덮지 않아 피해 숫자와 충격 연출이 남습니다.
- `resolveBattleOutcome`으로 결과 판정을 분리했으며, 보스와 성의 체력이 같은 틱에 0이 되면 마지막 아군 타격의 승리를 우선하는 기존 정책을 테스트로 고정했습니다.

### 거대 원반 소비와 발사 커밋

- `GameContext.consumeGiantDisc`는 렌더 전 projected 저장 상태에서 수량을 즉시 한 개 줄여 같은 재고를 연속 입력이 중복 소비하지 못하게 합니다.
- 엔진은 최신 전투 상태로 발사 후보를 먼저 계산하고, 저장 수량 소비가 승인된 경우에만 후보를 전투 상태에 커밋합니다.
- `BattleScreen`은 소비와 발사를 따로 조합하지 않고 `throwGiantDisc`의 단일 성공 결과만 처리합니다.
- 발사 불가 상태는 소비 함수를 호출하지 않고, 재고가 없으면 유효한 발사 후보도 폐기합니다.

### 전투 완료 상태·결과 단일 전이

- `completeBattleTransition`은 전투 승리 한 건에 대한 다음 `GameState`와 `BattleRewardResult`를 한 번에 계산하는 순수 함수입니다.
- `GameContext.completeBattle`은 같은 전이를 즉시 `stateRef`에 투영해 렌더 전에 연속 완료 요청이 와도 다음 스테이지를 기준으로 결과를 반환합니다.
- `gameReducer`의 `COMPLETE_BATTLE`도 같은 전이 함수의 상태를 사용하므로 UI의 최초 보상·스테이지·해금 표시와 저장되는 승리 수·거대 원반·보상 기록이 서로 다른 규칙으로 계산되지 않습니다.
- 잠긴 난이도나 알 수 없는 난이도의 완료 요청은 상태를 그대로 유지하고 보상을 반환하지 않습니다.
- 최초 스테이지 보상, 렌더 전 연속 완료, 20번째 승리 해금, 클리어 스테이지 재도전, 잘못된 난이도 요청을 회귀 테스트로 고정했습니다.
- 승리 상태가 먼저 렌더되고 영구 보상 결과가 effect에서 확정되는 짧은 구간에는 결과 모달이 `보상 확인 중`을 표시하고 이동 버튼을 잠급니다. 보상이 없는 재도전으로 잘못 표시되거나 완료 전 다음 전투를 시작하지 않습니다.

### JSON 설정 로딩 경계 검증

- `validateGameConfig`가 앱 시작 시 로드하는 23개 게임 JSON 설정의 필수 필드, 중첩 타입, 유한한 숫자, ID·레벨 중복을 외부 라이브러리 없이 검사합니다. 별도 `navigation.json`도 전용 모델이 시작 시 검증합니다.
- 처음 작성한 692줄 단일 검증 파일은 책임별 모듈로 분리했습니다. 공개 진입점 `validation/index.ts`(75줄), 공통 검증 도구 `primitives.ts`(176줄), 전투 `combat.ts`(220줄), 성장·상점 `economy.ts`(108줄), 시스템·표현 `system.ts`(282줄), 교차 참조 `references.ts`(104줄), 타입 계약 `types.ts`(77줄)로 구성되며 순환 의존은 없습니다.
- 난이도→웨이브→몬스터, 난이도→맵, 난이도→적 원반과 저장 마이그레이션 별칭의 대상 ID를 교차 검증합니다. 오류에는 `DIFFICULTIES[0].enemyWaveId`처럼 정확한 설정 경로가 포함됩니다.
- `getDifficulty`, `getBattleMapForDifficulty`, `getEnemyDisc`, `getMonster`, `getEnemyWave`, `getCookie`는 알 수 없는 키를 첫 행으로 대체하지 않고 명시적인 `ConfigValidationError`를 냅니다. 구매 명령에서 선택적으로 조회하는 `getUpgrade`, `getBot`, `getDisc`는 기존 반환 타입대로 `undefined`를 반환합니다.
- 모든 명시 레벨 테이블은 `1..N` 연속성을 요구합니다. 전투 tick·이동 divisor·공격 주기·원반 쿨타임·애니메이션 분모는 `0`보다 커야 하지만, 단계별 보너스·추가 적 수·최초 공격 지연처럼 `0`이 의미 있는 필드는 계속 허용합니다.
- 보스 강공격 진행률은 `0 < windup < slam < recovery < 1`이어야 합니다. 전장 시작·종료점, 원근 위치, 적 렌더 최소·기준·최대 크기와 망치 충격 반지름처럼 계산에 필요한 최소 의미 순서도 검증합니다.
- 런타임이 안전 정수로 처리하는 강화 값·가격·레벨·크기·속도·쿨타임·몬스터 기준 HP/공격력은 설정 로딩 때도 `Number.isSafeInteger` 범위의 정수만 허용합니다. 난이도·쿠키·봇·몬스터 배율과 비율은 유효한 소수 값을 그대로 보존합니다.
- 전투 플래시 최대 불투명도를 포함한 `BattleFeedbackConfig`의 모든 필수 필드를 검증하며, 불투명도·정규화 좌표·HUD 너비·원근 위치·그림자 비율·분노 체력 비율·거대 원반 화면 비율과 HSL 범위를 의미 있는 상한 안으로 제한합니다. 피격 burst의 좌표·지연·크기도 렌더 전에 검증합니다.
- 효과음 설정은 레벨이 정확히 `[1,2,3,4,5]` 순서이고 실제 음량이 단계마다 커야 합니다. 저장 스키마 버전도 `save-migrations.json.currentSaveVersion`의 양의 정수에서 읽습니다.
- 필드 누락, 잘못된 중첩 타입, 중복 ID, 끊어진 참조·마이그레이션 대상, 잘못된 조회 키, 0인 분모·주기, 레벨 누락과 잘못된 의미 순서를 회귀 테스트로 고정했습니다.

### 상점 ID 조회와 명령 거부 계약

- `getDiscProgress(state)` 무인자 호출은 손상된 선택 ID를 복구하기 위해 첫 원반 fallback을 유지합니다. 반면 `getDiscProgress(state, discId)`처럼 명시한 ID가 잘못되면 `undefined`를 반환하며 다른 원반으로 바꾸지 않습니다.
- `BUY_DISC`와 `UPGRADE_DISC` reducer는 명시 ID 조회 실패 시 원래 상태 객체를 그대로 반환합니다. `getUpgradeProgress`·`getBotOffer` 및 대응 구매 reducer도 같은 거부 계약을 유지합니다.
- projected command 계층은 잘못된 원반·강화·쿠키봇 ID에 모두 `false`를 반환하고 상태 참조와 dispatch 목록을 바꾸지 않는 것을 회귀 테스트로 고정했습니다.

### 저장 경계 정규화

- 외부 저장 숫자는 공통 정책으로 유한성을 확인하고, 내림해 정수화한 뒤 필드별 최소·최대 범위로 제한합니다. 일반 수량·레벨·시각은 `Number.MAX_SAFE_INTEGER`, 난이도 승리 수는 설정된 스테이지 수를 상한으로 사용합니다.
- 쿠키·누적 쿠키, 쿠키 강화·원반 레벨, 쿠키봇·거대 원반 보유 수량, 난이도 승리 수, 음량 단계, 저장 시각과 이전 버전 단일 원반 레벨에 같은 경계를 적용했습니다. 이전·현재 쿠키봇 수량 합산도 안전 정수 상한에서 포화됩니다.
- `soundEnabled`·`vibrationEnabled`는 실제 불리언만 허용하고, 클리어 난이도 목록은 알려진 난이도의 정규화된 승리 수에서 재구성합니다. 누적 쿠키는 현재 쿠키 이상이라는 불변식을 유지하며, 잘못된 타입의 저장 배열은 빈 배열로 복구합니다.
- `NaN`, 양·음의 `Infinity`, 음수, 소수, 과도한 유한 숫자, 손상된 배열·불리언과 저장 불변식을 회귀 테스트로 고정했습니다.
- 현재 앱보다 높은 `saveVersion`은 알려진 필드만 메모리에서 정규화하고 오프라인 정산·debounce 저장·백그라운드 저장을 모두 중지합니다. 구버전 실행 중 진행은 영구 반영하지 않는 대신 최신 앱의 원본 저장을 다운그레이드가 덮어쓰지 않습니다.

### GameProvider 책임 분리와 명령 직렬화

- 249줄이던 `GameContext.tsx`는 62줄의 조정·context 제공 계층으로 줄었습니다. Provider는 reducer와 최신 상태 참조를 만들고 아래 hook을 조합한 뒤 값을 제공하는 역할만 합니다.
- `useGamePersistence`는 저장 불러오기·오프라인 생산 정산, 미래 버전 읽기 전용 판정, debounce 저장, 비활성 AppState 전환 시 최신 projected 상태 저장을 담당합니다. 기존 저장 키와 `HYDRATE` 형식은 바꾸지 않았습니다.
- `useAutoProduction`은 실행 중 생산 타이머를 소유합니다. 완성된 생산 주기만 기준 시각을 이동시키고 남은 밀리초를 다음 틱으로 넘기는 계산은 `calculateAutoProductionTick` 순수 함수로 분리했습니다.
- Provider가 만드는 중앙 `createProjectedGameDispatcher`를 `useGameCommands`와 `useAutoProduction`이 함께 사용합니다. 사용자 입력과 같은 틱의 자동 생산도 현재 `stateRef`에 reducer를 먼저 적용하고, 실제 변경된 경우에만 참조 갱신과 dispatch를 수행합니다.
- `useGameCommands`는 클릭·구매·장착·난이도·도감·설정·전투 완료·거대 원반 명령 생성만 담당합니다.
- `completeBattle`은 `completeBattleTransition`의 결과와 미리 계산한 상태를 같은 projected 경계에 전달합니다. 따라서 UI 반환 결과, 렌더 전 연속 전투 진행과 reducer 저장 상태가 계속 일치합니다.
- 정확히 1회 강화 비용으로 빠르게 두 번 누르면 `[true, false]`, 2단계 비용이면 단계별 가격으로 `[true, true, false]`가 되는 동작과 클릭·설정·도감·전투 완료·거대 원반 연속 명령을 회귀 테스트로 고정했습니다.

- 415줄이던 `state/gameReducer.ts`는 137줄 액션 전이 파일로 줄었습니다. 초기 상태는 `gameInitialState.ts`(34줄), 외부 저장 정규화·이전·오프라인 복구·미래 버전 보호는 `gameSave.ts`(278줄)가 담당하며 기존 공개 export는 모두 유지합니다.
- 계산·저장·액션이 각각 독립되어 새 쿠키, 무기, 봇, 난이도 또는 저장 이전 규칙을 추가할 때 관련 책임 파일만 수정할 수 있습니다.

### 데이터 기반 4대 메뉴

- 하단 메뉴는 게임·전투·강화·정보 네 대분류입니다. 강화는 쿠키·원반·쿠키봇 강화, 정보는 쿠키 도감·난이도·몬스터 소메뉴를 직접 열며 게임·전투에는 소메뉴 View가 없어 전투 공간을 줄이지 않습니다.
- `navigation.json`이 순서, 한국어 라벨, 아이콘, 제목, 기본 leaf와 배지 키를 관리합니다. 전용 검증은 모든 화면의 정확한 1회 소속, 기본 leaf 참조와 중복 ID를 확인합니다.
- 앱 실행 중 대분류별 마지막 leaf를 기억하고 몬스터 NEW 배지는 몬스터 소메뉴와 정보 대분류에 함께 표시됩니다.

### 런타임 숫자 포화와 강화 상한 경계

- `safeNumbers.ts`가 `0..Number.MAX_SAFE_INTEGER` 정규화, 포화 덧셈·뺄셈, 선형·곱셈·지수 파생값과 안전한 다음 레벨 계산을 공통 제공하며 `NaN`·`Infinity`·음수 입력을 거부합니다.
- 쿠키 획득·구매, 쿠키봇 수량, 전투 거대 원반 보상·소비와 오프라인 생산 정산은 모두 포화 연산을 사용합니다. 정상 액션 뒤 쿠키·누적 쿠키·보유 수량·레벨은 항상 안전한 음이 아닌 정수이고 누적 쿠키는 현재 쿠키 이상입니다.
- 쿠키·원반 무한 강화와 쿠키봇 가격은 큰 레벨에서도 value·damage·size·speed·cost·cooldown을 유한한 안전 값으로 제한합니다. `MAX_SAFE_INTEGER` 레벨·수량에서는 `next`와 구매 가능 상태가 사라져 비용만 차감되는 전이가 없습니다.
- 쿠키 진화 총 레벨, 쿠키 배율 능력치, 쿠키봇 총 수량과 전투 난이도 파생값도 포화·유한 경계를 사용합니다. 정상 범위의 기존 수식과 테이블 밸런스는 유지합니다.
- `lastSavedAt`이 손상된 포화 최댓값이면 hydrate 시 현재 시각으로 체크포인트를 복구해 이후 자동 생산이 재개됩니다. 일반적인 짧은 기기 시계 역행은 기존처럼 이전 체크포인트를 보존하고 보상을 만들지 않습니다.
- 극단값 helper, MAX 직전·MAX 강화, 보상·소비, 경제·전투 selector, 중앙 projected 자동 획득→구매 순서와 미래 체크포인트 복구를 별도 회귀 테스트로 고정했습니다.

### 전투 엔진 수치 포화

- `battleNumbers.ts`가 여러 전투 배율의 기존 계산 순서와 마지막 `round`·`ceil` 위치를 보존하면서, 큰 유한 곱·나눗셈·합계만 `Number.MAX_SAFE_INTEGER`에서 포화합니다. 따라서 정상 범위의 보스 체력과 DPS·원반 피해 밸런스는 기존 결과와 같습니다.
- 성·쿠키봇·거대 원반 피해, 쿠키봇 군단 DPS, 보스 체력과 일격 방지 체력, 적 원반·근접 공격 및 여러 적 원반의 합산 피해에 같은 경계를 적용했습니다.
- 플레이어·적 투사체의 레벨·피해·크기·속도와 전투 시작 성 체력을 안전 범위로 제한하고, 보스·성 HP와 처치 수는 포화 차감·덧셈을 사용합니다. 최대 원반 능력치와 최대 쿠키봇 수량을 함께 사용해도 전투 상태에 `Infinity`나 안전 범위를 넘는 숫자가 남지 않습니다.
- MAX 원반·쿠키봇 기반 계산, 플레이어·적 투사체 생성, 여러 최대 피해 합산, 보스·성 체력 차감을 회귀 테스트로 고정하고 정상 군단 DPS의 기존 부동소수점 결과도 정확히 일치하는지 확인합니다.

### 전투 엔진 책임 분리와 공용 공격 selector

- 926줄이던 `useBattleEngine.ts`를 199줄의 상태·타이머·명령 조정 hook으로 줄였습니다. 기존 export는 facade에서 재노출해 테스트와 화면의 import 경로를 유지합니다.

### P3 — 계획적으로 줄일 구조·성능 부채

1. 현재 단일 보스·종류별 봇 합산 구조에서는 틱당 배열 복사와 선형 검색 비용이 작습니다. 향후 다수 적·상태 효과·다중 발사체를 다시 도입하면 매 50ms의 `map`, `filter`, `find`, 객체 복사가 개체 수에 따라 늘어납니다. 개체 수 확장 전에 프로파일링 기준과 상한을 정하고, 필요할 때만 인덱스 또는 object pool을 도입해야 합니다.

## 검증 현황과 빈틈

리뷰 수정 후 TypeScript 검사, Jest 10개 스위트 190개 테스트, `git diff --check`가 통과했습니다. 테스트는 데이터 타입·참조·의미·안전 정수/소수 계약, 5단계 음량, 메뉴 소속·기억·배지, 잘못된 상점 ID 거부, 단일 보스 전투, 공용 보스 공격 타이밍 selector, 원반 사거리·피해와 극단 전투 수치 포화, 빠른 성 공격·게임 명령 직렬화, 거대 원반 소비·발사 원자성, 수동 투사체 ID, 음수 프레임 방어, 보스 행동, 보상·저장 이전·미래 버전 보호와 오프라인 생산 시계를 포함합니다.
