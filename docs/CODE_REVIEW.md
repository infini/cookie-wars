# 코드 리뷰 기록

## 리뷰 정보

- 기준일: 2026-07-18
- 기준 버전: 1.0.8
- 범위: 전투 입력·시뮬레이션·결과 처리, 영구 상태와 저장, JSON 설정 계층, 전투 화면 구성, 테스트와 확장성
- 심각도 기준: P1은 즉시 배포를 막아야 하는 치명 문제, P2는 다음 기능 확장 전에 해결할 정확성 문제, P3는 계획적으로 줄일 구조·성능 부채입니다.

## 결론

P1 문제는 발견되지 않았습니다. 현재 한 마리 보스 전투, 난이도별 맵·보스 선택, 전투 화면 탭 공격과 저장 흐름은 기존 테스트 범위에서 동작합니다. `BattleScreen`의 표현 책임도 분리되어 새 연출을 추가할 때 한 파일을 계속 키우지 않아도 되는 상태가 되었습니다.

다만 “모든 함수와 파일이 이미 단일 책임을 만족한다”고 판단할 단계는 아닙니다. 동일 프레임 이벤트를 하나만 보존하는 방식 등은 같은 틱의 복합 사건이나 상태 시스템 사이에서 불일치가 생길 여지가 있습니다. 다음 대규모 전투 기능을 추가하기 전에 아래 P2를 우선 해결해야 합니다.

## 이번에 완료한 구조 개선

### 전투 화면 분리

기존 1,064줄 `src/screens/BattleScreen.tsx`를 511줄의 화면 조정 계층으로 줄이고, 표시 책임을 다음 네 파일로 옮겼습니다.

| 파일 | 줄 수 | 책임 |
|---|---:|---|
| `src/components/battle/BattleHealthBar.tsx` | 52 | 체력 비율·색상과 게이지 렌더링 |
| `src/components/battle/BattleImpactEffect.tsx` | 204 | 피격 파편·충격파·피해 숫자 렌더링 |
| `src/components/battle/BattleProjectiles.tsx` | 168 | 아군·적군·거대 원반 발사체 렌더링 |
| `src/components/battle/BattleUnits.tsx` | 322 | 쿠키봇 편성, 보스 상태·공격 자세 렌더링 |

`BattleScreen`에는 전투 시작·종료, 영구 보상 연동, 효과음 명령, 화면 레이어 조합과 사용자 입력 조정이 남았습니다. 분리는 전투 판정식, 능력치, z-index와 터치 규칙을 바꾸지 않는 표현 계층 이동으로 진행했습니다.

### 난이도 콘텐츠 데이터화

- 15개 난이도는 `difficulties.json → enemy-waves.json → monsters.json`을 통해 각각 고유 보스 하나를 선택합니다.
- `battle-maps.json`은 각 난이도 ID와 고유 전장 하나를 연결합니다.
- 15종 보스의 기준 전투 프로필을 동일하게 유지해 에셋·이름 추가가 현재 밸런스를 바꾸지 않도록 했습니다.
- 정적 이미지 키 매핑은 `BattleMapImage.ts`와 `MonsterSprite.tsx`로 한정했습니다.
- 설정 테스트가 난이도별 맵·웨이브·보스 ID의 유일성과 참조 무결성을 확인합니다.

### 전투 화면 전체 탭과 전파 격리

활성 전투의 전장 `Pressable`만 성 원반 명령을 받습니다. 거대 원반 버튼과 쿠키 성 자체 입력은 `stopPropagation()`으로 상위 전장 탭과 분리되며, 거대 원반 버튼은 사용할 수 없는 상태에서도 이벤트 전파를 막습니다. 결과 모달과 하단 메뉴는 전장 입력 영역 밖에 있어 한 번의 탭이 전투 명령과 메뉴 명령을 동시에 만들지 않습니다.

엔진의 기존 쿨타임·사거리·소유 검증은 그대로 적용되므로 화면 빈 곳을 눌렀다는 이유만으로 조건을 우회하지 않습니다. 이 레이어 구성 자체에서 확인된 UI 부작용은 없습니다. 거대 원반 버튼도 엔진 명령 하나만 호출하고 상위 전장 탭과 분리됩니다.

### 엔진 입력 안정성 보강

- 타이머와 탭 명령을 최신 `stateRef`에서 직렬화하고, 순수 `tryThrowCastleDisc`·`tryThrowGiantDisc` 전이가 현재 상태의 전투 진행 여부·쿨타임·사거리를 다시 검증합니다.
- 같은 렌더 사이의 빠른 성 공격 두 번은 첫 발이 반영된 최신 쿨타임에서 두 번째 요청을 거부합니다.
- 수동 발사체 ID는 시각뿐 아니라 단조 증가 순번을 포함해 같은 밀리초 입력도 충돌하지 않습니다.
- 마지막 전달 이벤트의 ID와 객체를 기억해 `onEvent` 콜백 참조만 바뀐 경우 같은 효과음을 다시 전달하지 않습니다.
- 프레임 경과 시간은 `0~maxDeltaMs`로 제한해 기기 시각이 뒤로 이동해도 보스나 원반이 역행하지 않습니다.

이 보강은 전투 피해·쿨타임·속도·난이도 수치를 변경하지 않습니다.

### 거대 원반 소비와 발사 커밋

- `GameContext.consumeGiantDisc`는 렌더 전 projected 저장 상태에서 수량을 즉시 한 개 줄여 같은 재고를 연속 입력이 중복 소비하지 못하게 합니다.
- 엔진은 최신 전투 상태로 발사 후보를 먼저 계산하고, 저장 수량 소비가 승인된 경우에만 후보를 전투 상태에 커밋합니다.
- `BattleScreen`은 소비와 발사를 따로 조합하지 않고 `throwGiantDisc`의 단일 성공 결과만 처리합니다.
- 발사 불가 상태는 소비 함수를 호출하지 않고, 재고가 없으면 유효한 발사 후보도 폐기합니다.

## 남은 발견 사항

### P2 — 다음 전투 기능 확장 전에 해결

1. 전투 이벤트가 단일 슬롯입니다.

   `BattleState.lastEvent`는 한 프레임에서 우선순위가 가장 높은 이벤트 하나만 남깁니다. 마지막 타격과 승리가 같은 틱에 발생하면 `enemyDefeated`가 `victory`로 덮여 마지막 피격 이펙트·피해 숫자·효과음이 누락될 수 있습니다. 프레임 이벤트 배열 또는 제한된 큐를 만들고 단조 증가 ID로 정확히 한 번 소비해야 합니다.

2. 결과 예측 로직이 `GameContext`와 리듀서에 나뉘어 있습니다.

   `completeBattle`은 UI 반환용 최초 보상·해금 결과를 미리 계산하고, `gameReducer`는 영구 상태 변경을 다시 계산합니다. 두 식이 변경 때 어긋날 수 있으므로 하나의 순수 전투 완료 계산 함수가 “다음 상태 + 결과”를 함께 만들도록 통합하는 편이 안전합니다.

3. JSON 설정은 컴파일 타임 단언에 의존합니다.

   `src/config/index.ts`의 `as ...Config`는 APK 실행 시 잘못된 JSON 타입·누락 필드를 검증하지 않습니다. 일부 조회 함수의 첫 행 fallback도 잘못된 ID를 조용히 숨길 수 있습니다. 앱 시작 또는 테스트 빌드에서 스키마 검증 후 명확한 오류를 내는 경계가 필요합니다.

4. 저장 정규화의 숫자 검증이 모든 필드에 일관되지 않습니다.

   일부 레벨과 시각에는 `Number.isFinite`를 적용하지만 쿠키 수, 승리 수, 보유 수량 등 모든 외부 저장 숫자가 같은 검증을 거치는 것은 아닙니다. `NaN`, `Infinity`, 음수와 과도한 값을 필드별 정책으로 정규화하는 테스트를 추가해야 합니다.

### P3 — 계획적으로 줄일 구조·성능 부채

1. `src/engine/useBattleEngine.ts`는 안전한 수동 입력 전이를 포함해 880줄이고, 핵심 `advanceBattle`은 약 287줄입니다. 이동, 충돌, 봇 발사, 보스 원거리·근접 공격, 결과 판정을 순서대로 모두 담당합니다. 이벤트 큐 작업과 함께 `move`, `resolvePlayerHits`, `spawnBotProjectiles`, `resolveBossAttacks`, `resolveBattleResult` 같은 순수 단계로 나누고 수동 공격 명령도 별도 모듈로 옮기는 것이 좋습니다.

2. `BattleUnits`는 엔진 규칙을 읽어 공격 준비 시간과 자세를 UI에서 다시 계산합니다. 현재는 연출 전용이지만 엔진 공격 조건이 변하면 표시와 실제 판정이 어긋날 수 있습니다. 엔진이 표현용 phase/view-model을 제공하거나 공통 순수 selector를 두는 것이 안전합니다.

3. `GameProvider`는 저장 생명주기, 오프라인 생산, 도메인 명령, UI 반환 결과를 함께 조정합니다. 저장·시간 진행·구매/전투 명령을 각각 hook 또는 service 경계로 분리하면 기능 추가 때 회귀 범위를 줄일 수 있습니다.

4. 현재 단일 보스·종류별 봇 합산 구조에서는 틱당 배열 복사와 선형 검색 비용이 작습니다. 향후 다수 적·상태 효과·다중 발사체를 다시 도입하면 매 50ms의 `map`, `filter`, `find`, 객체 복사가 개체 수에 따라 늘어납니다. 개체 수 확장 전에 프로파일링 기준과 상한을 정하고, 필요할 때만 인덱스 또는 object pool을 도입해야 합니다.

5. 성과 보스가 같은 틱에 모두 0이 되는 경우 현재 처리 순서상 승리가 우선입니다. 의도한 게임 규칙을 문서로 결정하고 회귀 테스트를 추가해야 합니다.

## 검증 현황과 빈틈

리뷰 수정 후 TypeScript 검사, Jest 7개 스위트 67개 테스트, `git diff --check`가 통과했습니다. 테스트는 데이터 참조 무결성, 단일 보스 전투, 원반 사거리·피해, 빠른 성 공격 재검증, 거대 원반 소비·발사 원자성, 수동 투사체 ID, 음수 프레임 방어, 보스 행동, 보상·저장 이전과 오프라인 생산을 포함합니다.

현재 테스트만으로 보장하지 못하는 항목은 다음과 같습니다.

- 실제로 마운트한 hook에서 타이머 수명주기와 이벤트 중복 방지가 유지되는지
- 같은 틱에 발생하는 마지막 타격·승리·성 피해의 이벤트 순서
- 성과 보스의 동시 사망 결과 정책
- 잘못된 JSON 설정과 `NaN`·`Infinity`가 포함된 저장 데이터
- 실제 React Native `Pressable` 트리에서 중첩 탭 전파가 차단되는지

실기기 설치·콜드 스타트 성공은 실행 가능성 확인이며 장시간 프레임 성능이나 위 경쟁 조건을 증명하지는 않습니다. 성능 측정 기준은 `PERFORMANCE.md`를 따릅니다.

## 개선 우선순위

1. 이벤트 배열/큐와 정확히 한 번 전달 계약을 도입하고 같은 틱 복합 사건 테스트를 추가합니다.
2. `advanceBattle`을 순수 단계로 분해하면서 동시 승패 정책을 테스트합니다.
3. 전투 완료 계산을 하나의 순수 함수로 통합해 `GameContext`와 리듀서 중복을 제거합니다.
4. JSON 런타임 스키마와 저장 숫자 정규화 경계를 추가합니다.
5. 다수 적·상태 효과를 추가하기 직전에 동일 기기 기준 성능을 다시 측정하고 필요한 최적화만 적용합니다.
