# 코드 리뷰 기록

## 리뷰 정보

- 기준일: 2026-07-19
- 기준 버전: 1.0.17
- Android versionCode: 18
- 범위: 쿠키 성장·일반/슈퍼 크리티컬·쿠키 조각·확률 보정, 보스·쿠키봇 애니메이션, X1·X2·X3 전투 속도, 전투 판정, 저장 이전, JSON 검증, 이미지 레지스트리, 화면 렌더링과 성능 경계
- 심각도: P0/P1은 배포 차단, P2는 기능 정확성, P3는 확장 전 보강이 필요한 구조·성능 문제로 분류했습니다.

## 결론

최종 정적 리뷰에서 남은 P0·P1·P2·P3 결함은 발견하지 못했습니다. 리뷰 중 발견한 전투 타이밍, 표시 좌표, 배속 프레임, 데이터 계약과 도감 렌더링 문제는 모두 수정하고 회귀 테스트를 추가했습니다.

가장 큰 프로덕션 TypeScript 파일은 `GameScreen.tsx` 298줄, `gameSave.ts` 292줄이며 300줄을 넘는 실행 파일은 없습니다. 리뷰 중 326줄이 된 참조 검증에서 희귀 보상 교차 규칙을 `cookieSpecialReferences.ts` 220줄로 분리하고, 음량 단계 정규화를 `audioSettings.ts`로 옮겨 저장 복구와 사용자 설정 책임을 나눴습니다. 5초 조각 수명·수령, 특수 피드백 수명, 일반·특수 오디오, 조각 확률·보상은 각각 전용 훅·조정자·순수 도메인으로 분리했습니다. 화면은 상태를 표시하고 명령을 전달하며, 전투 판정은 `engine`, 성장 계산은 `domain`, 영구 상태는 `state`, 수치와 에셋 키는 `config`가 소유합니다.

## 리뷰 중 해결한 사항

### 보스 강공격과 실제 애니메이션

- 15종 보스는 각각 걷기 3프레임, 망치 준비·충돌·복귀 3프레임을 사용합니다. `BossAnimationSprite`는 현재 프레임 하나만 정적 `require`에서 표시합니다.
- `specialAttackCycleStartedAt`은 보스가 성 공격 사거리에 들어온 시각 또는 직전 강공격 시각을 보관합니다. 실제 충돌 시각인 `lastSpecialAttackAt`과 분리해 사거리 진입이 가짜 내려찍기 프레임으로 보이지 않게 했습니다.
- 강공격은 사거리 진입 후 정확히 5초마다 실행됩니다. 다음 강공격까지 남은 시간이 `현재 위치에서의 적 원반 예상 비행시간 + 900ms 준비 시간` 이내이면 일반탄 발사를 예약 차단합니다. 5초가 끝나면 일반 공격 쿨타임보다 `special`을 우선하므로 망치 준비 중 일반탄이 끼어들거나 충돌이 늦어지지 않습니다.
- 일반 원거리·근접 공격은 강한 망치 프레임을 재생하지 않습니다. 실제 `special`에서만 준비·충돌·복귀, 지면 균열, 먼지 충격파, 전장 흔들림과 플래시가 함께 나타납니다.

### 쿠키봇 이동·투척 좌표

- 5종 봇은 `bot-animation.json`의 순찰 반경·표적 추종·위상 차로 실제 전투 좌표를 달리며, 달리기 3프레임과 투척 3프레임을 공격 시각에 맞춰 표시합니다.
- 봇 원반의 논리 시작점은 발 좌표가 아니라 `projectileReleaseOffsetX/Y`를 더한 손 높이입니다. 사거리 판정, 발사체 원점, 전투 이벤트와 화면 windup도 같은 `getBotProjectileOrigin`·`isBotTargetInRange` 계산을 사용합니다.
- 투척 프레임 동안 표시 위치를 발사 순간에 고정해 봇이 달리는 중에도 손과 원반이 어긋나지 않습니다.

### X1·X2·X3 실제 전투 속도

- `battle-rules.json`이 `[1, 2, 3]`과 기본값을 관리하며, HUD 버튼은 X1 → X2 → X3 → X1로 순환합니다.
- 배속은 화면 숫자나 큰 델타만 바꾸지 않고 제한된 `advanceBattle` 서브스텝을 배수만큼 반복합니다. 이동, 쿨타임, 투사체, 충돌, 피해, 승패와 효과음 이벤트가 함께 빨라집니다.
- 선택 배속은 저장 스키마 v10에 포함되고 재실행·다음 전투에도 유지됩니다. 손상되거나 과거 버전의 값은 테이블에 정의된 배수로 정규화합니다.
- X3 첫 서브스텝에서 공격이 발생해도 마지막 렌더에서 핵심 충돌·투척 프레임이 남도록 보스 `impactHoldMs`와 봇 `throwReleaseHoldMs`를 250ms로 설정했습니다. 설정 검증은 `maxDeltaMs × (최대 배수 - 1)`보다 짧은 값을 거부합니다.

### 쿠키 크리티컬과 저장 안전성

- 일반 기본 확률은 정확히 1%, 기본 보상은 클릭 획득량의 10배입니다. 슈퍼 기본 확률은 0.1%, 기본 보상은 100배입니다. 네 희귀 보상 모두 200,000단위 정수로 계산해 슈퍼의 레벨당 0.025%p와 전기 조각의 0.0125%p를 오차 없이 표현합니다.
- 슈퍼 크리티컬은 Lv.1의 0.1%에서 레벨마다 정확히 50단위, 즉 0.025%p 증가합니다. 저장에는 확률값이 아니라 강화 레벨이 있으므로 기존 레벨도 별도 저장 마이그레이션 없이 새 테이블로 소급 계산됩니다. 예를 들어 Lv.20은 `0.1% + 19 × 0.025%p = 0.575%`입니다.
- 강화는 확률과 배수를 함께 올립니다. 확률은 50%에서 멈추고 보상 배수는 안전 정수 상한까지 계속 증가합니다.
- 클릭 한 번은 주입 가능한 난수로 순수 계산한 뒤 쿠키를 한 번만 지급합니다. 같은 크리티컬 여부와 지급량을 전체·축약 시각 피드백, 획득 텍스트와 Freesound·Mixkit 계층 사운드에 전달합니다.
- 3보이스 풀, 일반/슈퍼 계층 음원, 번개·천공 균열·다중 참격·각진 파편·화면 흔들림 보간 순서와 동시 이펙트 상한은 `cookie-feedback.json`과 전용 검증 모듈이 함께 관리합니다. 기존 동그란 링은 사용하지 않습니다. 사운드 끄기·설정 닫기·화면 전환은 예약된 계층 음원과 재생 중인 쿠키 소리를 취소합니다.
- 첫 실기기 점검에서 Expo Audio의 읽기 전용 `playbackRate` 속성 대입 오류를 발견해 공식 setter인 `setPlaybackRate` 호출로 교체했습니다. 최종 설치본에서는 일반 쿠키 클릭 경로를 실제 실행한 뒤에도 프로세스와 화면이 유지됐습니다.
- v9 이하 저장에는 새 강화의 기본 Lv.1을 채우고 v10부터 저장한 레벨과 배속을 복구합니다. 쿠키 수·강화 레벨·배수 계산은 `Number.MAX_SAFE_INTEGER`에서 포화되어 `Infinity`나 음수 상태를 만들지 않습니다.

### 쿠키 조각과 종류별 합성 피드백

- 마그마 0.2%와 전기 0.05%는 크리티컬과 독립된 한 난수를 겹치지 않는 구간으로 나눕니다. 두 조각은 동시에 발견되지 않고 Lv.1 보상은 수령 시점의 클릭 힘에 각각 50·200을 안전 정수 곱합니다. 강화마다 확률은 각 기본값의 25%, 배수는 10%만큼 같은 비율로 증가합니다.
- 클릭 명령은 발견 종류만 반환합니다. `useCookieFragmentCollection`이 활성 ID·종류를 일치시키고 5초 안의 첫 탭에만 별도 수령 명령을 보내므로, 만료·중복 탭·새 조각 교체는 쿠키를 지급하지 않습니다.
- 조각 `Pressable`은 메인 쿠키의 자식이 아니라 형제 오버레이여서 조각 탭으로 기본 클릭 보상이 함께 발생하지 않습니다. 등장 후 5초 수명과 타이머는 native-driver 진행값으로 움직이고 프레임마다 React 상태를 바꾸지 않습니다.
- `cookie-fragments.json`이 확률·배수·수명·조각 크기·부스러기·화산·하강 용암·다중 번개·색·음량을 소유합니다. 이미지 레지스트리는 두 수집 조각 키가 실제 투명 WebP에 연결되는지 검사하고, 설정 검증은 마지막 용암·번개도 페이드 전에 나타나는지 확인합니다.
- 일반·슈퍼·마그마·전기는 종류별 항목으로 최대 4개까지 함께 마운트합니다. 같은 종류가 다시 나오면 그 종류만 ID를 바꿔 처음부터 재생하고, 다른 종류는 유지합니다. 오디오도 동일한 종류별 플레이어·타이머 세대를 사용합니다.
- 전기는 이전 크랙·전자 폭발음을 번들에서 제거하고 Mixkit `Strong close thunder explosion`을 220ms 간격 세 번 재생합니다. 마그마는 생성한 쿠키 화산 이미지와 11줄기 용암·22개 불씨를 합성하며 두 조각 모두 화면 기준 크기로 확장됩니다.
- 시각 수명은 일반 930ms < 마그마 1,500ms < 슈퍼 1,800ms < 전기 1,950ms입니다. 음향 레이어 강도도 `1.14 < 1.84 < 1.98 < 3.0`으로 같은 순서를 따르며 교차 테이블 검증이 역전을 거부합니다. 슈퍼는 9분절 천공 균열·5중 참격·14갈래×5분절 번개·24개 파편으로 마그마보다 높은 시각 등급을 갖습니다.
- 저장 스키마는 슈퍼·두 조각 강화가 없는 이전 저장을 Lv.1로 채웁니다. 두 조각은 최고 레벨이 없고 확률 상한 뒤에도 배수가 성장합니다. 강화 화면에 보이는 7종은 모두 진화 기여 플래그가 `true`여서 기존 저장의 현재 단계까지 메인 쿠키 진화 레벨에 자동 소급됩니다.

### 희귀 보상 천장과 원자적 저장

- `cookiePity.ts`는 확률 눈금과 현재 확률 단위의 정수 나눗셈만으로 최대 시도 횟수를 계산하며 클릭당 O(1)입니다. 10% 일반 크리티컬의 10번째 확정, 기본 슈퍼·전기의 장기 천장과 동시 조각 우선순위를 순수 테스트로 검증했습니다.
- 자연 슈퍼는 일반 천장으로 낮아지지 않고 일반·슈퍼 카운터를 모두 초기화합니다. 전기와 마그마 천장이 겹치면 전기를 먼저 주고 마그마 카운터는 다음 클릭까지 유지합니다.
- `calculateCookieClickTransition`은 지급 결과와 다음 카운터를 한 객체로 만들고 `CLICK_COOKIE` 리듀서가 쿠키·누적 쿠키·카운터를 한 전이에서 커밋합니다. 저장 v13은 4종 카운터를 영구 복구하고 손상값을 안전 정수로 정규화합니다.

### 50종 쿠키 확장 계약

- 쿠키는 총 50종입니다. 요구 진화 레벨은 3, 9, 15, …, 297로 증가하고 세 공통 능력 배율은 1.00에서 62.06까지 완만하게 증가합니다.
- `validateCookies`는 요구 레벨의 엄격한 증가뿐 아니라 클릭·자동 생산·성 체력 배율의 동일성과 앞 쿠키보다 큰 공통 배율을 검증합니다. 화면이 세 값을 하나의 공통 배율로 표시하므로 테이블과 UX가 달라질 수 없습니다.
- `CookieImage`, `BossAnimationSprite`, `BotAnimationSprite`가 `has*Image` 레지스트리 검사를 공개합니다. `assetRegistry.test.ts`는 50개 쿠키, 90개 보스 프레임, 30개 봇 프레임의 모든 JSON 키가 실제 정적 이미지와 연결되는지 확인합니다.
- 쿠키 도감은 50개의 투명 WebP를 한꺼번에 마운트하지 않는 `FlatList`를 사용합니다. 화면 주변 카드만 렌더하므로 향후 쿠키를 더 추가해도 이미지 디코드 메모리가 종류 수에 정비례해 즉시 증가하지 않습니다.

### 데이터와 화면 책임

- 게임 결과에 영향을 주는 확률, 배수, 주기, 프레임 지속 시간, 순찰 거리와 발사 오프셋은 JSON 테이블에 있습니다. React Native 글자 크기·간격 같은 순수 표현값만 `StyleSheet`에 둡니다.
- `validation/animation.ts`가 프레임 배열, 참조 범위, 시간과 최대 배속 표시 가능성을 검증합니다. 기존 성장·상점·전투·저장 검증과 같은 앱 시작 경계에서 잘못된 테이블을 즉시 거부합니다.
- `BattleScreen`은 얇은 레이어 조정자이고, 이전의 `BattleUnits.tsx`는 공개 파사드만 담당합니다. 실제 봇·보스 표현은 `BattleBotFormation`과 `BattleEnemyLayer`, 프레임 표시는 별도 Sprite 컴포넌트가 담당합니다.
- 전투 엔진은 이동·아군 공격·보스 공격·배속 조정을 분리합니다. 가장 큰 관련 구현은 `enemyBattleSimulation.ts` 234줄, `useBattleEngine.ts` 211줄, `playerBattleSimulation.ts` 206줄입니다.

## 성능 검토

- 현재 전투는 보스 1개, 표시 봇 최대 5종, 설정된 상한의 적 발사체와 짧게 유지되는 이벤트 저널로 제한됩니다. 같은 종류의 봇 수량은 한 개 표시·한 발의 합산 피해로 처리합니다.
- X3은 50ms 타이머 한 번에 최대 3개의 제한된 서브스텝을 실행해 초당 최대 60회 순수 시뮬레이션 단계입니다. 큰 델타 한 번으로 충돌을 건너뛰지 않습니다.
- 보스 90개와 봇 30개 WebP는 모두 384×384이며 현재 상태의 프레임 하나만 렌더합니다. 쿠키 도감은 `FlatList` 가상화를 사용합니다.
- 정적 검토에서 뚜렷한 전투 병목은 발견하지 못했습니다. 다만 이번 버전의 장시간 FPS·메모리는 동일 기기에서 아직 다시 측정하지 않았으므로, 다수 적이나 상태 효과를 재도입하기 전에는 `PERFORMANCE.md` 절차로 실측해야 합니다.
